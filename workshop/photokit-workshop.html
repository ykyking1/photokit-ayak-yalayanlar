01 · Ortam Kurulumu — UV ~10 dk

UV nedir? Astral ekibinin yazdığı, pip + venv + poetry yerine geçen ultra-hızlı Python paket yöneticisi. Rust ile yazıldığı için pip'ten 10-100x daha hızlı. Tek komutla proje başlatır, bağımlılık ekler, lock dosyası oluşturur.

UV Kurulumu

macOS / Linux:

bashcurl -LsSf https://astral.sh/uv/install.sh | sh

Windows (PowerShell):

powershellpowershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

Doğrula:

bashuv --version

Temel UV Komutları

KomutAçıklama
uv init projemYeni bir Python projesi oluşturur (pyproject.toml + .venv)
uv add pillow clickDependency ekler ve yükler
uv add --dev pytest ruffSadece development bağımlılığı ekler
uv remove clickDependency siler
uv syncLock dosyasından tüm dependency'leri yükler
uv run python script.pyProjenin sanal ortamında script çalıştırır
uv lockuv.lock dosyasını günceller

Neden UV?

PhotoKit'te 5 farklı bağımlılığımız olacak (pillow, click, rich, pytest, ruff). UV hepsini lock'layarak versiyonları kilitler — herkeste aynı sürümler kurulur. Gerçek proje hayatında "bende çalışıyor" problemini bitirir.

Bu bölümde takımlar yok. Herkes kendi laptop'ında aynı egzersizleri yapar. Amaç: komutlara kas hafızası kazanmak.

Git'in Zihinsel Modeli

 Working Directory  ─(git add)─►  Staging Area  ─(git commit)─►  Local Repo  ─(git push)─►  Remote (GitHub)
    [düzenlenen]                   [hazırlanan]                    [commit'li]                  [paylaşılan]

Her dosya 4 durumdan birinde: untracked (git bilmez), modified (değişti), staged (add'lendi), committed (commit'lendi).

Egzersiz 4 — Lokal Conflict ve Çözümü

Workshop'ın en kritik kaslarından. Önce lokalde simüle ediyoruz.

bash# main'de notes.txt'nin içeriğini "apple" yap
echo "apple" > notes.txt
git add notes.txt && git commit -m "set note to apple"

# Yeni bir branch'te aynı satırı "pear" yap
git switch -c different-fruit
echo "pear" > notes.txt
git add notes.txt && git commit -m "set note to pear"

# main'e geç ve merge et — çakışacak!
git switch main
git merge different-fruit

Git şuna benzer bir mesaj verecek:

textAuto-merging notes.txt
CONFLICT (content): Merge conflict in notes.txt
Automatic merge failed; fix conflicts and then commit the result.

notes.txt'yi açtığında şunu göreceksin:

text<<<<<<< HEAD
apple
=======
pear
>>>>>>> different-fruit

Çözüm: Hangi versiyonu istediğine karar ver (veya ikisini birleştir), conflict marker'larını sil, sonra:

bashgit add notes.txt
git commit -m "resolve merge conflict: keep both fruits"

İpucu

VS Code, conflict'li dosyayı açtığında "Accept Current Change / Accept Incoming Change / Accept Both / Compare Changes" butonlarını gösterir. Görsel çözüm için çok pratik.

03 · Branch Nasıl Çalışır ~10 dk

Branch = bir commit'e pointer. Yeni commit yaptıkça pointer ileri kayar.

Şekil 1 — Basit Branch

gitGraph
   commit id: "A"
   commit id: "B"
   branch feat/blur
   checkout feat/blur
   commit id: "C"
   commit id: "D"
   checkout main
   commit id: "E"
   merge feat/blur
   commit id: "F"

feat/blur branch'i, main'den B commit'inde ayrıldı. Orada paralel 2 commit yapıldı (C, D). Bu sırada main'de E commit'i oldu. Sonunda merge ile birleşti.

Şekil 2 — Fast-Forward vs Three-Way Merge

Fast-forward: main, branch'ten beri ilerlememişse Git sadece pointer'ı ileri alır. Merge commit oluşmaz.

Three-way merge: main de ilerlediyse Git üç commit'i (ortak ata + iki uç) birleştirip yeni bir merge commit yaratır.

Şekil 3 — Merge vs Rebase

gitGraph
   commit id: "A"
   commit id: "B"
   branch feat/x
   commit id: "X1"
   commit id: "X2"
   checkout main
   commit id: "C"
   merge feat/x tag: "MERGE"

Merge her iki tarafı tutar, geçmişi gerçekçi gösterir. Rebase feat/x'teki commit'leri main'in ucuna kopyalar — doğrusal geçmiş elde edersin ama SHA'lar değişir.

Altın Kural

Başkasıyla paylaşılmış branch'e asla rebase yapma. Rebase sadece senin kendi lokal branch'lerin için güvenlidir.

Temel Branch Komutları

src/photokit/registry.py (Dikkat: Conflict Noktası)

pythonfrom .filters.base import Filter
from .filters.grayscale import GrayscaleFilter

# Her yeni filtre burada kaydedilmeli.
# Iki kisi ayni anda filtre ekledigi zaman CONFLICT cikar!
FILTERS: dict[str, type[Filter]] = {
    "grayscale": GrayscaleFilter,
}


def get_filter(name: str) -> Filter:
    if name not in FILTERS:
        available = ", ".join(FILTERS.keys())
        raise ValueError(f"Unknown filter: {name!r}. Available: {available}")
    return FILTERS[name]()

src/photokit/cli.py

pythonimport click
from PIL import Image
from rich.console import Console
from .registry import get_filter, FILTERS

console = Console()


@click.group()
@click.version_option()
def main() -> None:
    """PhotoKit — apply filters to images from the command line."""
    pass


@main.command("list")
def list_filters() -> None:
    """List all available filters."""
    console.print("[bold cyan]Available filters:[/]")
    for name in FILTERS:
        console.print(f"  - {name}")


@main.command()
@click.argument("name")
@click.argument("input_path", type=click.Path(exists=True, dir_okay=False))
@click.argument("output_path", type=click.Path(dir_okay=False))
@click.option("--radius", type=int, default=2, help="Blur radius (for blur filter)")
@click.option("--width", type=int, default=800, help="Target width (for resize filter)")
def apply(name: str, input_path: str, output_path: str, radius: int, width: int) -> None:
    """Apply a filter to an image."""
    try:
        img = Image.open(input_path)
        flt = get_filter(name)
        out = flt.apply(img, radius=radius, width=width)
        out.save(output_path)
        console.print(f"[green]\u2713[/] {name} applied \u2192 {output_path}")
    except ValueError as e:
        console.print(f"[red]Error:[/] {e}")
        raise SystemExit(1)


if __name__ == "__main__":
    main()

pythonfrom PIL import Image
from .base import Filter


class SepiaFilter(Filter):
    name = "sepia"

    def apply(self, img: Image.Image, **kwargs) -> Image.Image:
        img = img.convert("RGB")
        pixels = img.load()
        width, height = img.size
        for py in range(height):
            for px in range(width):
                r, g, b = pixels[px, py]
                tr = int(0.393 * r + 0.769 * g + 0.189 * b)
                tg = int(0.349 * r + 0.686 * g + 0.168 * b)
                tb = int(0.272 * r + 0.534 * g + 0.131 * b)
                pixels[px, py] = (min(255, tr), min(255, tg), min(255, tb))
        return img

pythonfrom PIL import Image, ImageOps
from .base import Filter


class InvertFilter(Filter):
    name = "invert"

    def apply(self, img: Image.Image, **kwargs) -> Image.Image:
        # ImageOps.invert RGBA'yi desteklemez, alpha kanalini koruyarak tersle
        if img.mode == "RGBA":
            r, g, b, a = img.split()
            rgb = Image.merge("RGB", (r, g, b))
            inverted = ImageOps.invert(rgb)
            r2, g2, b2 = inverted.split()
            return Image.merge("RGBA", (r2, g2, b2, a))
        return ImageOps.invert(img.convert("RGB"))

pythonfrom PIL import Image
from .base import Filter


class ResizeFilter(Filter):
    name = "resize"

    def apply(self, img: Image.Image, **kwargs) -> Image.Image:
        width = kwargs.get("width", 800)
        if width <= 0:
            raise ValueError("Width must be a positive integer")

        # Aspect ratio'yu koruyarak yeniden boyutlandir
        ratio = width / img.width
        height = int(img.height * ratio)
        return img.resize((width, height), Image.Resampling.LANCZOS)

İyi Commit Mesajı Yazmak

Commit mesajı sadece sana değil — 6 ay sonra o satırı görüp "bu neden böyle?" diye git blame açan kişiye yazdığın bir açıklamadır. İyi commit mesajı, kodun tarihçesi olur.

Bir Commit Mesajının Anatomisi

textfeat: add blur filter with configurable radius

Implements Gaussian blur using Pillow's ImageFilter.GaussianBlur.
Users can control intensity via the --radius CLI flag (default: 2).

Also registers the new filter in registry.py. Future filter PRs
(sepia, invert) will need to update the same file — expect
merge conflicts.

Closes #1

Üç bölüm:

  • 1. satır (subject): Conventional Commit prefix'iyle başlar, emir kipinde, ≤50 karakter, sonunda nokta yok.
  • Boş satır: Subject ile body arasında olmak zorunda. Git subject'i buradan ayırır.
  • Body (opsiyonel ama önerilir): Ne ve Neden — "nasıl"ı kod zaten söylüyor. 72 karakterde satır kaydır.

Emir Kipi (Imperative Mood) Kuralı

Git'in kendi konvansiyonu. İngilizce yazacağız (kural), ve "If applied, this commit will ______" cümlesini tamamlayacak şekilde yaz:

Yanlış (Past / Continuous)Doğru (Imperative)
feat: added blur filterfeat: add blur filter
fix: fixing resize bugfix: handle zero-width in resize filter
docs: updated readmedocs: update readme with new filter list
chore: version bump to 1.1chore: bump version to 1.1

Dil Kuralı

Bu workshop'ta ve takım repolarında: kod ve commit mesajları İngilizce, workshop anlatımı ve sözlü tartışma Türkçe. Commit tarihçesi açık kaynak dünyasının ortak dili İngilizce olduğu için bu standarttır — CI bot'ları, changelog araçları, release-notes üreticileri hepsi İngilizce bekler.

Küçük PR, Mutlu Takım

PR'lar 400 satırın altında olsun. Araştırmalar gösteriyor ki 400 satırı aşan PR'larda bug yakalama oranı belirgin şekilde düşüyor. Büyük PR'ların sorunları:

  • Review edilmez (okuyan yorulur → onay rubber-stamp olur)
  • Geç merge olur → conflict riski büyür
  • Bir şey kırılırsa git bisect zorlaşır
  • Revert gerekirse yan hasar çıkar

Büyük bir feature geliyorsa parçala: birbirine bağımlı 3 küçük PR, tek bir devasa PR'dan her zaman iyidir.

Anti-pattern

PR açarken "bu arada şunu da temizledim" diye 5 ayrı refactor eklemeyin. Her PR tek bir işe odaklanmalı. Refactor ayrı PR, feature ayrı PR, formatting ayrı PR.

Agent-Team Pattern (Senin Kullanım Şeklin)

Birden fazla AI agent (veya paralel iş) aynı repo üzerinde çalıştığında her agent'a bir worktree ver:

Worktree Kuralları

  • Aynı branch iki worktree'de checkout edilemez (Git koruması).
  • Her worktree ayrı bir klasör — editörde ayrı window aç.
  • Silme = git worktree remove, rm -rf değil (metadata kalır).
  • .git klasörü sadece ana worktree'de bulunur, diğerleri .git dosyası (file, not folder) içerir.

Yap

Her takım üyesi kendi aktif branch'indeyken bir takım arkadaşının branch'i için worktree oluştursun, oradan bir küçük test yapsın, sonra worktree'yi kaldırsın.

13 · Kurtarma Toolkit'i ~10 dk

Git nadiren kod kaybeder. Commit edilmiş her şey reflog'da saklanır (90 gün varsayılan).

Çalıştığım Değişiklikleri Geçici Sakla

bash# Degisiklikleri stash'le (gecici koy)
git stash

# Stash'leri listele
git stash list

# En son stash'i geri al
git stash pop

# Specific stash'i al
git stash apply stash@{1}

# Stash'leri temizle
git stash drop stash@{0}
git stash clear  # hepsi

Son Commit'i Geri Almak

KomutNe Yapar?
git commit --amendSon commit'i değiştirir (push etmediysen güvenli)
git reset --soft HEAD~1Commit'i iptal et, değişiklikler staged kalsın
git reset --mixed HEAD~1Commit'i iptal et, değişiklikler working'de kalsın (varsayılan)
git reset --hard HEAD~1Commit'i ve değişiklikleri sil (tehlikeli)
git revert HEADCommit'in tersini yapan yeni commit — paylaşılmış kodda bu

Reset vs Revert

Revert history'i korur, yeni bir "geri alma" commit'i üretir — paylaşılmış branch'lerde kullan. Reset history'i değiştirir — sadece kendi lokal branch'inde kullan.

Reflog — "Bir commit'i sildim, geri dönebilir miyim?"

bash# Son hareketleri gor (silinmis commit'ler bile)
git reflog

# Orneğin "HEAD@{3}" diye bir satir gorursun
# O commit'e don:
git reset --hard HEAD@{3}

Reflog sayesinde hard reset bile geri dönülebilirdir (commit'ler 90 gün saklanır).

Yanlış Branch'e Commit Attım

bash# main'de olmaman gereken bir commit'in var
git log --oneline       # son commit'in SHA'si: abc1234

# Commit'i main'den kaldir (ama kaybetme)
git reset --soft HEAD~1

# Dogru branch'e gec
git switch -c feat/correct-branch

# Commit'i o branch'te yap
git commit -m "feat: move changes to correct branch"

Sık Yapılan Hatalar

  • Secret commit etmek — API key, şifre, vs. Çare: git filter-repo + key'i iptal et.
  • Devasa PR açmak — Review edilmez. Küçük, bağımsız PR'lar.
  • "fix typo" gibi gürültü commit'leri — Interactive rebase ile squash et: git rebase -i HEAD~5.
  • Shared branch'e force push — Başkasının işini siler. Asla.
  • node_modules / .venv commit etmek — .gitignore unutulmuş demek.
